Class Diagram

UML class diagram of all domain entities. User is the shared abstract account base — both Community Member and Water Service Provider can register, log in, verify their email, and reset their password the same way, so those fields/methods live once on User and are inherited rather than duplicated. There is no admin account type and no role field anywhere — which kind of account it is, is simply which subclass it is. Water Service Provider adds organization-level attributes (registration number, service categories, availability) and can employ ProviderStaff, cover ProviderServiceAreas, and handle/attend the Reports, Service Requests, and Appointments that a Community Member submits or books. Each entity carries its own specifically named identifier (userId, reportId, …) rather than a generic id; audit timestamps (createdAt/updatedAt) are omitted from the diagram for clarity, though they still exist in the database via SQLAlchemy's TimestampMixin. Enumeration types (status, category, urgency, audience, etc.) are kept as plain attributes rather than separate classes to keep the diagram focused on structure and behaviour — their full value sets are documented in the ERD and README Section 11. Report, ServiceRequest, and Appointment inherit shared SLA-tracking behaviour (status, escalationLevel, escalate()) from the abstract Trackable class. Location is a reusable value object for Rwanda's 5-level administrative hierarchy (province → district → sector → cell → village), embedded wherever an address is needed. Visibility markers: + public  |  - private. Multiplicity is shown on all associations. Composition (◆⎯) denotes lifecycle dependency; aggregation (◇⎯) denotes shared reference.

Entity Relationship Diagram (ERD)

Simplified ERD showing the 8 core PostgreSQL entities and the 10 foreign-key relationships that connect them. Each entity lists only its primary key, foreign keys, and essential business columns — internal fields (hashed_password, SLA-tracking timestamps, coordinates, audit timestamps) are omitted for readability. Supporting tables (PROVIDER_STAFF, PROVIDER_SERVICE_AREAS, REPORT_MEDIA) exist in the database but are excluded from this diagram to keep the structure easy to follow — they are documented in full in README Section 11. USSD_SESSION (dashed lines) represents the stateless USSD channel — Africa's Talking manages session state; Temba decodes navigation from the text callback parameter and authenticates via USERS.ussd_pin_hash. All primary keys are UUID.

System Architecture Diagram

Six-layer modular service architecture, organized by responsibility for clarity: 1. Client Layer (Community Member Portal, Provider Dashboard, USSD feature phones) → 2. Network & Security Layer (SSL/TLS, NGINX Load Balancer, API Gateway, Rate Limiting) → 3. Application Layer (nine FastAPI business services — Authentication, Reporting, Service Requests, Appointments, Provider Management, Announcements, Notifications, Analytics, USSD Handler — plus a lightweight Navigation Chatbot that guides users around the platform's features without resolving issues itself) → 4. Accountability & SLA Layer (Response Monitoring, Automated Reminders, 4-level Escalation, Community Verification, Audit Logging) → 5. Data Layer (PostgreSQL 16, Redis 7, MinIO S3 Object Storage) → 6. External Integrations (Africa's Talking USSD gateway, Gmail SMTP, SMS notifications). There is no admin portal — admin operations are performed directly via the API.
